# StyTrix Techpack 網站架構圖

範圍：線上網站（前端 + Vercel Functions + GitHub Actions pipeline + 外部服務 + 資料相依）的全貌。
只畫「跑在線上或會被線上 trigger 到」的東西；純線下一次性的 `scripts/` 規則產線不在本圖內（細節見 `pom_rules_pipeline_guide_v2.md`）。

---

## 1. 高階流程（Browser → Vercel → 外部 API / Actions）

```mermaid
flowchart LR
  U([使用者瀏覽器])
  Admin([Admin 使用者<br/>需 x-admin-token])

  subgraph Vercel["Vercel（stytrix-techpack.vercel.app）"]
    direction TB
    SS["Static 託管<br/>index.html · LOGO.png<br/>data/ · pom_rules/ · l2_l3_ie/<br/>l1_*.json · General Model_Path2/"]
    F1["api/analyze.js<br/>(Node, maxDuration 300s)"]
    F2["api/push-pom-dict.js<br/>(Node, maxDuration 30s)"]
    F3["api/ingest_token.js<br/>(Node, maxDuration 10s)"]
    F4["api/ingest_upload.js<br/>(Node, maxDuration 60s)"]
  end

  CV[(Anthropic<br/>Claude Vision API)]
  GH[(GitHub Contents API<br/>ellyfd/stytrix-techpack)]

  subgraph Actions["GitHub Actions<br/>.github/workflows/rebuild_master.yml"]
    A1["push 到 data/ingest/uploads/**<br/>觸發 rebuild"]
  end

  U -- "GET /<br/>載入 SPA + JSON 資料" --> SS
  U -- "POST /api/analyze<br/>(image + context + mode)" --> F1

  Admin -- "POST /api/push-pom-dict<br/>翻譯表編輯器" --> F2
  Admin -- "POST /api/ingest_token<br/>取 GitHub PAT" --> F3
  Admin -- "POST /api/ingest_upload<br/>(≤4.5MB PDF/PPTX)" --> F4
  Admin -. "直連 GitHub（>4.5MB 或 JSON patch）<br/>用拿到的 PAT 直接 PUT contents" .-> GH

  F1 -- "messages.create<br/>Pass 1 + Pass 2" --> CV
  F2 -- "PUT data/pom_dictionary.json" --> GH
  F4 -- "PUT data/ingest/uploads/<file>" --> GH

  GH -. "push 事件觸發" .-> A1
  A1 -- "commit rebuilt data/" --> GH
  GH -. "[skip ci] commit 觸發<br/>Vercel redeploy" .-> Vercel
```

**關鍵設計**：
- 純靜態 SPA（無 bundler、無 `package.json`），React 走 CDN，整個 app 在 `index.html` 內聯。
- 所有 functions 都是 Vercel **Node.js runtime**，不是 Edge runtime。
- `api/analyze.js` 在 module init 階段 `fs.readFileSync` 讀 `data/l2_visual_guide.json` 與 `data/l2_decision_trees.json`，靠 `vercel.json` 的 `includeFiles: "data/**"` 把檔打包進 function bundle（先前改成 fetch 會 hang 到 timeout）。
- `api/ingest_token` 的作用只有一個：把 `GITHUB_PAT` 給已驗證的 admin，讓瀏覽器**直連** GitHub API，繞過 Vercel 4.5MB body 上限（PPTX 常超過）。較小檔或伺服端控制的情境才走 `api/ingest_upload`。
- 資料重建**不經過 Vercel function**——是 GitHub Actions workflow 在 GitHub 端跑的。

## 4. Ingest Pipeline（PATH2 Phase 2：PDF/PPTX → 自動重建 recipes_master）

使用者丟一份新 Techpack 進 UI，到 `data/recipes_master.json` 更新、Vercel 重新部署，**完全不需要本機環境**。

```mermaid
flowchart TB
  Uploader([Admin 在前端<br/>拖一份 PDF/PPTX 上傳])

  subgraph BrowserPath["上傳路徑（選一）"]
    direction LR
    BP1["檔案 ≤ 4.5 MB<br/>→ /api/ingest_upload<br/>(Vercel 代 commit)"]
    BP2["檔案 > 4.5 MB<br/>→ /api/ingest_token 取 PAT<br/>→ 瀏覽器直連 GitHub API"]
  end

  GH1[("GitHub repo<br/>data/ingest/uploads/&lt;file&gt;")]

  subgraph WF["GitHub Actions：rebuild_master.yml<br/>trigger: push paths data/ingest/uploads/**"]
    direction TB
    S1["Step 1 extract_raw_text.py<br/>掃 uploads/，D-number 去重<br/>→ metadata/designs.jsonl (append)<br/>→ pptx/*.txt<br/>→ pdf/callout_images/*.png<br/>→ pdf/callout_manifest.jsonl"]
    S2a["Step 2a extract_unified.py<br/>合併 4 來源 (PPTX 中文 / PDF 英文 / cb / dir5)<br/>→ data/ingest/unified/{dim,facts}.jsonl"]
    S2b["Step 2b vlm_pipeline.py<br/>(需 ANTHROPIC_API_KEY secret)<br/>→ data/ingest/vlm_v1/<br/>continue-on-error"]
    S3["Step 3 build_recipes_master.py --strict<br/>→ data/recipes_master.json<br/>→ data/iso_dictionary.json<br/>→ data/l1_standard_38.json"]
    S4["git rm uploads/*<br/>+ commit 重建結果<br/>commit msg 含 [skip ci]"]
    S1 --> S2a --> S2b --> S3 --> S4
  end

  GH2[("GitHub main<br/>data/recipes_master.json 等")]
  V1["Vercel 重新部署<br/>前端吃到新 recipes"]

  Uploader --> BrowserPath
  BP1 --> GH1
  BP2 --> GH1
  GH1 -- "push 事件" --> WF
  S4 -- "git push origin HEAD" --> GH2
  GH2 -. "push webhook" .-> V1
```

**細節**：
- workflow 只在 `data/ingest/uploads/**` 有變動時觸發，其他地方改動不會啟動整條 pipeline。
- Step 2b 失敗不會擋 build（`continue-on-error: true`），因為 VLM 只是增補訊號。
- Step 3 用 `--strict`，任何 ingest/taxonomy 違規就 exit 1，**不**把壞資料 push 回去。
- workflow commit 帶 `[skip ci]`，避免重建結果再觸發自己。
- `extract_unified.py` 是**全量重建**（吃所有歷史 facts），不是增量 append，所以輸出 deterministically 一致。

## 5. Admin 通道總表

| 端點 | 用途 | 身份驗證 | Body 大小 |
|---|---|---|---|
| `POST /api/push-pom-dict` | 前端翻譯表編輯器寫回 `data/pom_dictionary.json` | `x-admin-token` | 小 |
| `POST /api/ingest_token` | 取回 `GITHUB_PAT` 給瀏覽器直連 GitHub | `x-admin-token` | — |
| `POST /api/ingest_upload` | PDF/PPTX 經 Vercel 代 commit 到 `data/ingest/uploads/` | `x-admin-token` | ≤ 4.5 MB |
| 瀏覽器直連 GitHub API（JSON patch / 大檔） | 用 `ingest_token` 拿到的 PAT 直接 `PUT /repos/.../contents/<path>` | 瀏覽器帶 `Authorization: token <PAT>` | 無實質上限 |

**環境變數**（Vercel Project Settings）：
- `ANTHROPIC_API_KEY` — `api/analyze.js` 呼叫 Claude Vision
- `ADMIN_TOKEN` — 所有 admin 端點的共享 token
- `GITHUB_PAT` — 寫回 GitHub 用的 PAT（contents: write）

GitHub Actions secrets（repo settings）：
- `ANTHROPIC_API_KEY` — Step 2b VLM pipeline 用
- `GITHUB_TOKEN`（自動注入）— Step 4 push 用

## 7. 已知架構債

- `recipes/`（根目錄 72 檔）目前無人引用；PATH2 規劃用的是 `construction_recipes/`（不同名）。下輪清理候選，須先跑 CLAUDE.md Part B 的 grep gate。
- `l1_part_presence_v1.json` / `l1_iso_recommendations_v1.json` 還在 repo 根，不在 `data/`。移動會動到 `index.html` 裡兩處 fetch path。
- `iso_lookup_factory_v4.json` 已經是 fallback，但仍被前端引用（提供 iso_zh / 機種），不能刪。
- `api/analyze.js` 的 `GUIDE` / `TREES` 是 function init 載入；改 `data/l2_visual_guide.json` 或 `l2_decision_trees.json` 後，需要 Vercel redeploy 才會生效（不是熱更新）。
- Ingest workflow 的 Step 2b (`vlm_pipeline.py`) 是 `continue-on-error`——失敗時會安靜跳過，不會在 PR / commit 訊息裡浮現，需要直接翻 Actions log 才看得到。
- `scripts/`（線下規則產線）與 `star_schema/scripts/`（線上 ingest pipeline）是**兩組獨立**的 pipeline，彼此不共用程式碼，也沒有統一入口；動其中一邊不會自動影響另一邊。
